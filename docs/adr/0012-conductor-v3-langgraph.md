# ADR 0012 — Conductor v3 on LangGraph (4-mode topology)

Date: 2026-05-23
Status: accepted
Supersedes: ADR 0009 (list interpreter), ADR 0011 (framework evaluation deferred)

## Context

ADR 0009 landed a hand-rolled list interpreter for v2. Two days later (ADR 0011) we evaluated LangGraph and pipecat-flows as alternatives, deferred the migration, and shipped v2 to internal use. Within hours of v2 going live, dogfeeding revealed two structural gaps:

1. **No "follow-up control" knob for researchers.** v2 always probed when slots were missing. There was no way for a researcher to say "this is a structured questionnaire, do not probe" or "this is a deep-dive, probe aggressively." Real research designs need both.
2. **The list interpreter was carrying weight that paid back nothing.** v2 had typed `SlotSchema` (5 types), `SkipIfRule` (6 operators), `BehaviorSlots` (5 dimensions), per-turn `SessionEvent` event sourcing — none of which the in-flight outlines actually used. The cost was complexity (~3000 LOC + 150 tests) without measured benefit.

The decision: rebuild on LangGraph, drop the unused capability surface, and bake follow-up mode into the graph topology as a first-class routing key. ADR 0009 is superseded; ADR 0011's deferral is also superseded (it documented LangGraph as an option; we now take it).

## Decision

### Architecture

Conductor v3 is a fixed 5-node LangGraph `StateGraph`:

```
START → ask → [judge_off | judge_standard | judge_deep] → advance | ask
                                                            ↓
                                                       ask | END
```

- **Outline is data, not graph topology.** Researchers write a list of questions; the graph traverses it.
- **3 follow-up modes are first-class.** Each mode is its own judge node; the routing function `route_after_ask` dispatches based on `state["follow_up_mode"]`.
- **`probe_instruction` is researcher-written prose, used verbatim.** The judge prompt embeds it as the probing tactic guideline. No session-start LLM rewriting.
- **No finalize node.** Final-report generation is delegated to the existing post-session pipeline (`merism.conductor.post_session` + `SessionInsight` agents) reading `InterviewSession.transcript`.
- **`interrupt()` + `Command(resume=...)` for human-in-the-loop.** Voice and HTTP both drive the same graph; the wrapper layer handles transport.

### What v3 keeps from v2

- Researcher mental model: list of sections × list of questions
- Question fields: `id / ask / follow_up_mode / probe_instruction` (4 total — `id` auto-generated by frontend; researchers write 3)
- 2-LLM-call-per-turn budget envelope (Rule 4 reframing — see below)
- Rule 12 — AI is content-only; flow control is rule-driven
- Rule 13 — dual-engine routing (v1 stays on `stream_turn`, v3 via `is_v3_session(session)`)
- `team_id` multi-tenant scoping (data layer)

### What v3 drops from v2 (explicitly out of scope)

| Dropped | Rationale |
|---|---|
| Typed `SlotSchema` (5 types + enum_values) | Free-form prose `probe_instruction` is enough as judge guidance |
| Per-question `must_get: list[str]` and `goal: str` | Researchers don't write these — they only write `ask` + (optional) `probe_instruction`. Judge evaluates "did the user respond to the question, given probe_instruction?" |
| Per-question `max_followups: int` | The follow-up budget is determined by `follow_up_mode` (off=0, standard=2, deep=4), configured globally via `Configuration.standard_followups` / `deep_followups`. Researchers don't set per-question counts. |
| `SkipIfRule` cross-question jumps (6 operators) | Linear traversal only; conditional branching needs an ADR |
| `BehaviorSlots` (refused/evasive/off_topic/pre_answered/garbled) | Folded into `judge.sufficient` boolean |
| Per-turn `SessionEvent` as runtime authority | LangGraph checkpointer is the runtime authority; final transcript at `InterviewSession.transcript` (Rule 9 relaxed below) |
| Question rewriting at session start | Question text is verbatim |
| **`prepare_session` node** (probe_instruction expansion) | `probe_instruction` is researcher-written prose, used verbatim by the judge prompt — no LLM rewriting needed; saves 1 LLM call/session |
| **`finish_interview` node** (final-report generation) | Existing `merism.conductor.post_session` + `SessionInsight` pipeline already produces reports asynchronously, with cleaning + structured insight extraction — duplicating it inside the graph is wasteful |
| Visual graph editor for researchers | Researchers write a list, not a graph |
| `migrate_guide_to_v2` command | No data to migrate (0 v2 sessions ever ran in production) |

### Three follow-up modes

| Mode | Judge LLM call/turn | Behavior | Probe budget per question |
|---|---|---|---|
| `off` | 0 | Skip judge entirely; advance after each user reply | 0 (no probes) |
| `standard` | 1 | Lenient sufficiency bar; probe when answer doesn't address main question | `Configuration.standard_followups` (default 2) |
| `deep` | 1 | Strict bar (concrete detail required); probes more aggressively | `Configuration.deep_followups` (default 4) |

Mode is set **per question** via `Question.follow_up_mode` (researchers
mark background questions `off` and pain-point deep-dives `deep`).
Per-mode budget lives in `Configuration` — researchers do NOT set
follow-up counts per question; the mode picks the budget.

### Rule 4 reframing (AGENTS.md update)

The pre-v3 Rule 4 said "every user turn runs 2 sequential LLM calls." v1 split was extract→generate; v2 was the same. v3 collapses the two calls into a single judge call that returns both `sufficient: bool` and `followup: str | None` in one structured Pydantic response.

Updated invariant:
- v3 per-turn LLM calls: **at most 1** (the judge); **0** in `off` mode
- v3 per-session LLM calls: **0** outside the turn loop. Final-report generation is delegated to the existing post-session pipeline (`merism.conductor.post_session` + `SessionInsight` agents) which reads `InterviewSession.transcript`.
- v1 per-turn LLM calls: still 2 (decide + generate, unchanged)

Total envelope per turn is preserved or reduced in v3. AGENTS.md Rule 4 will be reworded to allow either pattern depending on engine.

### Rule 9 relaxation

Rule 9 said "event sourcing is authoritative; caches can be rebuilt." For v3:

- `merism_session_event` is **not** the runtime authority
- LangGraph's `PostgresSaver` checkpoint is the runtime authority
- Final transcript + report land on `InterviewSession.transcript` + `moderator_state.final_report` once at session end
- Existing analytics / report code reads `InterviewSession.transcript` unchanged

This trade gives us LangGraph's battle-tested resume/replay semantics in exchange for losing per-turn event-level granularity for v3 sessions. v1 sessions retain full event sourcing.

If we need event-level analytics for v3 later, a checkpoint→event mirror sink can be added without changing the engine.

### LLM client constraints (locked by Phase 0 spike)

- **Method**: `llm.with_structured_output(Schema, method="json_mode")`. The default `json_schema` raises HTTP 400 against DeepSeek (50/50 calls in spike). Function calling works but only at 84% well-formed. `json_mode` was 100%.
- **Prompt**: every template that drives `with_structured_output` MUST contain the literal word `JSON`; DeepSeek rejects otherwise (HTTP 400 safety check).
- **Temperature**: judge nodes default to 0.0 (deterministic); finalize at 0.3.

### File layout (binding)

```
merism/conductor_v3/
├── schema.py            # Outline / Section / Question / Turn (Pydantic)
├── state.py             # OverallState / JudgeOutput / AdvanceOutput (TypedDicts)
├── tools_and_schemas.py # Evaluation (LLM contract — only one Pydantic schema in v3)
├── prompts.py           # 2 module-level templates (judge_standard + judge_deep)
├── configuration.py     # Configuration BaseModel + from_runnable_config
├── llm.py               # build_llm + build_evaluator (json_mode)
├── nodes.py             # 5 nodes: ask_and_wait / judge_off / judge_standard / judge_deep / advance_cursor
├── graph.py             # build_graph + 3 routing functions
├── runner.py            # start_interview / answer_interview / get_interrupt_payload
├── persistence.py       # finalize_to_session — bridge to InterviewSession.transcript
├── router.py            # is_v3_session (Rule 13)
├── text_adapter.py      # HTTP turn dispatcher (start vs answer)
└── tests/
```

This layout is borrowed from `google-gemini/gemini-fullstack-langgraph-quickstart` (file split, state decomposition, configuration loading, structured-output Pydantic). Pattern provenance is documented in `docs/specs/conductor-v3/design.md` §0 as a binding contract.

## Consequences

### Positive

- Researcher gets a real follow-up mode toggle (off / standard / deep) — primary use case finally addressed
- 4× less code than v2: ~600 LOC v3 vs ~3000 LOC v2
- LangGraph PostgresSaver replaces our hand-rolled `event_log` + `persistence` modules
- Resume / replay / time-travel debugging available out of the box
- Future "concept testing" and other complex per-question patterns can be added as subgraphs without changing the outer 7-node topology
- DeepSeek + LangChain integration verified in Phase 0 spike (100% well-formed responses, p95 1.78s)

### Negative

- LangChain dependency footprint (~40 transitive deps including `langchain-core`, `langgraph-checkpoint`, etc.). Pinned in pyproject.toml.
- Rule 9 relaxation: v3 sessions don't write per-turn `SessionEvent`s. Event-level analytics for v3 needs a future bridge if requested.
- Frontend OutlineTab needs a separate v3 editor surface (typed slots / skip_if controls removed from UI).
- ADR 0009 is superseded only 2 days after acceptance — second framework swap in a month. Engineering cost is real, but the v2 capability surface was unused so there's nothing to throw away.

### Neutral

- ADR 0011's deferral is replaced. The triggers it specified (≥2 of: research-design gaps, time-travel debugging request, team growth, new requirements) had all fired by 2026-05-23.
- No data migration: 0 v2 sessions ran in production before v2 was deleted.
- AGENTS.md Rules 4, 9, 13 are all amended; specific edits committed alongside this ADR.

## Triggers to revisit

- Researchers ask for cross-question branching (`skip_if` equivalent) → write ADR 0013 extending `route_after_advance`
- Multi-stimulus / concept testing requires per-question multi-step subflow → write ADR adding subgraph for that question type
- DeepSeek introduces strict-mode JSON schema support → reconsider `method="json_schema"` for type safety wins
- Self-host LangSmith for telemetry → add tracing wiring (not in scope for v3 ship)
